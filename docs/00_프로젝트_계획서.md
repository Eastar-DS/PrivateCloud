## 1. 사업 개요

### 1.1 추진 배경 및 필요성

A기관 내 각 부서는 업무 관련 문서를 개별적으로 관리하고 있으며, 부서 간 문서 공유 및 검색에 어려움을 겪고 있다. 특히 과거 문서들은 체계적인 분류 없이 보관되어 있어 필요한 자료를 찾는 데 많은 시간이 소요된다.

또한 문서 공유 시 접근 권한 설정이 복잡하여, 담당자들이 "전체 공개"로 설정하는 경우가 빈번하게 발생하고 있어 보안 취약점이 존재한다.

이에 AI 기반 자동 태깅 기능을 갖춘 문서 스토리지 시스템을 구축하고, 이를 안전하게 운영할 수 있는 멀티 테넌트 프라이빗 클라우드 인프라가 필요하다.

### 1.2 서비스 목적 및 목표

에이블스택(ABLESTACK) HCI 기반 프라이빗 클라우드 위에 AI 문서 태깅 시스템을 구축하여, A기관 내 3개 부서(부서 A, 부서 B, 부서 C)가 문서를 안전하게 저장·공유·검색할 수 있는 환경을 제공한다.

#### 핵심 목표

| 구분 | 목표 |
|------|------|
| **서비스** | AI 기반 문서 자동 태깅 및 검색 시스템 구축 |
| **인프라** | 부서 간 데이터 격리가 가능한 멀티 테넌트 클라우드 구축, 공용으로 이용할 문서 자동 태깅 시스템 인프라 구축 |
| **보안** | 문서별 접근 권한 관리 및 부서 간 데이터 유출 방지 |

### 1.3 사업 범위

#### 대상 기관
- A기관 (3개 부서)

#### 서비스 범위

| 구분 | 내용 |
|------|------|
| **문서 저장** | 웹 기반 문서 업로드 및 다운로드 |
| **자동 태깅** | AI 모델을 통한 문서 주제 자동 분류 |
| **검색** | 태그 및 내용 기반 문서 검색 |
| **접근 제어** | 부서별 / 문서별 권한 설정 |
| **OCR** | 스캔 문서 및 이미지 내 텍스트 추출 |

#### 인프라 범위

| 구분 | 내용 |
|------|------|
| **테넌트 구성** | 부서 단위 테넌트 구성 |
| **인프라 구축** | 프라이빗 클라우드 기반의 통합 인프라 구축 |
| **데이터 격리** | 테넌트 간 데이터 논리적 격리 |
| **관리 체계** | 통합 관리 및 모니터링 체계 제공 |

#### 제외 범위

| 구분 | 내용 |
|------|------|
| **외부망 연동** | 외부 기관 또는 외부 시스템과의 연동은 본 프로젝트 범위에서 제외 |
| **데이터 이관** | 기존 시스템의 문서 데이터 마이그레이션은 수행하지 않음 |
| **하드웨어 장비 도입** | 신규 서버·스토리지·네트워크 장비 구매 및 설치는 제외 |
| **대규모 AI 학습** | 대규모 GPU 기반 모델 학습 및 성능 고도화는 제외 |

### 1.4 기대 효과

| 기대 효과 | 내용 |
|----------|------|
| **문서 검색 시간 단축** | AI 태깅으로 원하는 문서를 빠르게 검색 |
| **보안 강화** | 부서별 격리 + 문서별 접근 권한으로 정보 유출 방지 |
| **업무 효율성 향상** | 과거 문서 활용도 증가, 중복 작업 감소 |
| **비용 절감** | 통합 인프라로 부서별 개별 구축 대비 비용 절감 |
| **확장성 확보** | 부서/문서 증가 시 유연한 확장 가능 |

## 2. 현행 업무 분석 및 문제점

### 2.1 현행 업무 현황

A기관은 현재 부서별로 문서를 개별 관리하고 있으며, 공유가 필요한 문서는 별도의 공용 시스템에 업로드하고 있다.

| 부서 | 현황 |
|------|------|
| **부서 A** | 자체 파일 서버에 문서 보관 |
| **부서 B** | 외부 클라우드 스토리지 사용 |
| **부서 C** | 공용 시스템 + 개별 PC 혼용 |

#### 공용 문서 시스템 현황
- 문서 업로드 시 수동으로 분류/태깅 필요
- 과거 문서는 연도별로만 정리되어 있음
- 검색 기능 미흡 (파일명 검색만 가능)

### 2.2 문제점 분석

#### 1) 문서 검색의 어려움
- 과거 문서들이 체계적으로 분류되어 있지 않음
- 필요한 자료를 찾기 위해 수기로 뒤져야 함
- 검색 시 파일명만 검색 가능, 내용 검색 불가

#### 2) 접근 권한 관리 미흡
- 문서 업로드 시 접근 권한 설정이 복잡함
- "어디까지 공개해야 할지 몰라서" 전체 공개로 설정하는 경우 빈번
- 민감 문서가 불필요하게 노출될 위험

#### 3) 부서 간 데이터 격리 부재
- 공용 시스템에서 부서 간 데이터가 혼재
- 타 부서 문서에 접근 가능한 경우 존재

#### 4) 분산된 인프라
- 부서별 개별 스토리지 운영으로 비용 낭비
- 통합 관리 불가

### 2.3 개선 방향

| 문제점 | 개선 방향 | 적용 기술 |
|--------|----------|----------|
| 문서 검색 어려움 | AI 기반 자동 태깅 | OCR + AI 태깅 모델 |
| 접근 권한 관리 미흡 | 문서별/부서별 권한 체계 | ACL, 테넌트 기반 접근 제어 |
| 부서 간 격리 부재 | 테넌트별 스토리지 분리 | Pool 분리, CephX, Quota |
| 분산된 인프라 | 통합 프라이빗 클라우드 | HCI 기반 통합 인프라 |

## 3. 사업 추진 계획

### 3.1 추진 체계

본 프로젝트는 인프라 설계 및 구축, 서비스 개발을 중심으로 네트워크/보안, 컴퓨팅, 스토리지, 서비스 개발 파트로 나누어 추진한다.

```

│                      프로젝트 추진 체계                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐                                          │
│   │  PM / PL    │                                          │
│   └──────┬──────┘                                          │
│          │                                                  │
│   ┌──────┴──────────────────────────────────────┐          │
│   │                                              │          │
│   ▼                                              ▼          │
│   ┌───────────────────────┐    ┌───────────────────────┐   │
│   │     인프라 파트        │    │    서비스 파트         │   │
│   ├───────────────────────┤    ├───────────────────────┤   │
│   │ • 네트워크/보안        │    │ • 백엔드 개발         │   │
│   │ • 컴퓨트              │    │ • 프론트엔드 개발      │   │
│   │ • 스토리지            │    │ • AI 모델 연동         │   │
│   └───────────────────────┘    └───────────────────────┘   │

```

### 3.2 단계별 이행 전략

| 구분 | 내용 |
|------|------|
| **1단계** | 인프라 기반 구성 |
| **2단계** | 프라이빗 클라우드 코어 구성 |
| **3단계** | 테넌트 구성 및 서비스 연동 |
