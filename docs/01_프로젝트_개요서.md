## 1. 사업 개요

### 1.1 추진 배경 및 필요성

#### 1.1.1 정책 환경

2026년 재정경제부는 공공기관 인프라 활용 정책을 **개별 기관 중심에서 협업·확산 체계로 전환**한다고 발표했다.

| 정책 방향 | 세부 내용 |
|----------|----------|
| **공동 구축·활용** | 기관별 분산된 인프라를 통합하여 중복 투자 방지 |
| **표준화 추진** | 공공기관에 적합한 표준 아키텍처 정립 필요 |
| **데이터 주권 강화** | 민감 데이터의 외부 유출 방지를 위한 프라이빗 클라우드 인프라 필요성 대두 |

#### 1.1.2 기술적 문제

기관별 개별 인프라 구축으로 인해 예산 낭비 및 유휴 자원이 발생되고 있으며, 인프라 구축 역량이 부족한 중소 공공기관의 경우 안정된 보안 환경 조성에 어려움을 겪고 있다.

다기관 환경에서 프라이빗 클라우드를 운영하려면 다음의 인프라 설계 과제가 존재한다:

| 과제 | 설명 |
|------|------|
| **멀티테넌시** | 여러 기관이 자원을 공유하면서도 완전한 격리 보장 |
| **고가용성(HA)** | 단일 장애점(SPOF) 제거, 서비스 연속성 확보 |
| **망분리** | 공공기관 보안 요건에 따른 다계층 네트워크 분리 |
| **스토리지 효율화** | 대용량 데이터의 티어링, 백업, DR 설계 |

#### 1.2.2 주요 기술 트렌드

| 기술 영역 | 트렌드 | 본 프로젝트 적용 |
|----------|--------|----------------|
| **SDN** | 네트워크의 소프트웨어 정의화 | VLAN 기반 망분리, 마이크로세그멘테이션 |
| **SDS** | 분산 스토리지 기술 | Ceph 기반 분산 스토리지, 티어링 |
| **IaC** | 인프라 코드화 | Terraform/Ansible 자동화 |
| **Zero Trust** | 신뢰하지 않고 항상 검증 | RBAC, 마이크로세그멘테이션 |

### 1.3 서비스 내용

에이블스택(ABLESTACK) HCI 기술 기반의 **멀티테넌트 프라이빗 클라우드 플랫폼**으로, 여러 기관이 각자 독립된 가상 환경에서 안전하게 인프라를 이용할 수 있도록 설계한다.

#### 1.3.1 목표 서비스: 공용 AI 플랫폼

본 인프라가 지원해야 할 목표 서비스는 **공용 AI 플랫폼**이다. 이 플랫폼은 4계층 구조로 운영된다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    목표 서비스: 4계층 AI 플랫폼 구조                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   [계층 1: 테넌트별 격리 환경]                                              │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │   테넌트 A (A시청)      테넌트 B (B구청)      테넌트 C (C군청)       │  │
│   │   ┌───────────────┐    ┌───────────────┐    ┌───────────────┐       │  │
│   │   │ 민원 데이터    │    │ 민원 데이터    │    │ 민원 데이터    │       │  │
│   │   │ 테넌트 전용    │    │ 테넌트 전용    │    │ 테넌트 전용    │       │  │
│   │   │ 특화 모델     │    │ 특화 모델     │    │ 특화 모델     │       │  │
│   │   └───────┬───────┘    └───────┬───────┘    └───────┬───────┘       │  │
│   └───────────┼────────────────────┼────────────────────┼───────────────┘  │
│               └────────────────────┼────────────────────┘                  │
│                                    ▼                                        │
│   [계층 2: 데이터 품질 게이트] ─ 검증된 데이터만 공용 모델에 반영            │
│                                    │                                        │
│                                    ▼                                        │
│   [계층 3: 공용 AI 플랫폼] ─ 공용 모델 학습/서빙, GPU 풀 관리               │
│                                    │                                        │
│                                    ▼                                        │
│   [계층 4: 서비스 계층] ─ 테넌트 특화 모델 우선, 공용 모델 폴백             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 1.3.2 인프라 요구사항 도출

위 목표 서비스를 지원하기 위해 인프라가 충족해야 할 요구사항:

| 목표 서비스 요구 | 인프라 요구사항 | 설계 영역 |
|----------------|----------------|----------|
| 테넌트별 데이터 격리 | **멀티테넌시**: 네트워크/스토리지/컴퓨트 격리 | 전 영역 |
| 테넌트 간 데이터 혼입 방지 | **망분리**: VLAN 분리, 마이크로세그멘테이션 | 네트워크 |
| 대용량 학습 데이터 처리 | **스토리지 티어링**: Hot/Warm/Cold 자동 이동 | 스토리지 |
| 서비스 연속성 | **고가용성(HA)**: VM 자동 복구, 이중화 | 컴퓨트 |
| 장애 시 데이터 보호 | **DR/백업**: 스냅샷, 복제, 복구 계획 | 스토리지 |
| 민감 데이터 보호 | **보안**: Zero Trust, RBAC, 암호화 | 보안 |
| 운영 효율성 | **자동화**: IaC, 모니터링, 로깅 | 운영 |

#### 1.3.3 프로젝트 핵심 목표

> **HCI 기반 프라이빗 클라우드에서 다기관 환경을 안전하게 수용하는 인프라 아키텍처를 설계하고 구축한다.**

본 프로젝트는 **인프라 설계 역량**을 검증하는 것이 핵심이다:

```
┌─────────────────────────────────────────────────────────────────┐
│                     프로젝트 비중                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ████████████████████████████████████████░░░░░░░░░░           │
│   │                                      │          │           │
│   │         인프라 설계 (80%)            │ 검증(20%)│           │
│   │                                      │          │           │
│   │  - 멀티테넌시 격리 구조              │ 샘플     │           │
│   │  - 망분리 및 보안 아키텍처           │ 워크로드 │           │
│   │  - HA/DR 설계                        │ 기반     │           │
│   │  - 스토리지 티어링                   │ 인프라   │           │
│   │  - 운영 자동화 (IaC)                 │ 동작검증 │           │
│   │                                      │          │           │
└─────────────────────────────────────────────────────────────────┘
```

> **참고**: AI 플랫폼 애플리케이션 자체의 구현(모델 학습, 품질 게이트 등)은 본 프로젝트 범위가 아니며, 인프라가 이러한 서비스를 **지원할 수 있는 기반**을 설계하는 것이 목표이다.

#### 1.3.4 목표 상세

| 구분 | 목표 | 검증 방법 |
|------|------|----------|
| **멀티테넌시** | 3개 테넌트 완전 격리 | 테넌트 간 접근 불가 테스트 |
| **고가용성** | VM 장애 시 자동 복구 | Failover 시연 |
| **망분리** | 6개 이상 네트워크 존 분리 | 트래픽 격리 검증 |
| **스토리지** | Hot/Warm/Cold 티어링 | 데이터 이동 자동화 |
| **보안** | Zero Trust 아키텍처 | RBAC, 마이크로세그멘테이션 |
| **운영** | IaC 기반 자동화 | Terraform/Ansible 배포 |

### 1.4 사업 범위

#### 1.4.1 In-Scope (수행 범위)

**인프라 설계 (핵심)**

| 영역 | 세부 항목 |
|------|----------|
| **컴퓨트** | 장애 도메인 설계, VM 배치 전략, 리소스 풀링, 리소스 쿼터 |
| **네트워크** | 망분리 설계, VLAN 구성, 마이크로세그멘테이션, SDN 정책, 로드밸런서 |
| **스토리지** | 분산 스토리지 설계, 티어링 정책, 스냅샷/백업, DR 복제 |
| **보안** | Zero Trust 아키텍처, RBAC, 암호화 체계 |
| **운영** | IaC(Terraform/Ansible), 모니터링(Prometheus/Grafana), 로깅 |

**검증 워크로드 (부차적)**

| 항목 | 목적 |
|------|------|
| 샘플 애플리케이션 | 인프라의 **멀티테넌시 격리, 고가용성, 네트워크 분리**를 검증하기 위한 워크로드 |

#### 1.4.2 Out-Scope (제외 범위)

- 애플리케이션 개발 또는 고도화
- 실제 대국민 서비스 제공
- 퍼블릭 클라우드 또는 하이브리드 구성
- 상용 운영 환경 전환
- GPU 하드웨어 실물 구성

## 2. 현행 업무 분석 및 문제점

### 2.1 현행 업무 현황

공공기관 인프라 정책이 개별 기관 중심에서 협업·확산 체계로 전환됨에 따라, 기관별 분산형 인프라의 통합 필요성이 대두되고 있다.

```

│                                                                             │
│   기관 A              기관 B              기관 C              기관 D        │
```

### 2.2 현행 시스템 및 정보화 현황

| 구분 | 현황 | 문제점 |
|------|------|--------|
| **인프라 구축** | 기관별 개별 구축 | 중복 투자, 높은 단가 |
| **자원 활용** | 평균 활용률 20~30% | 유휴 자원 과다 |
| **보안 체계** | 기관별 상이한 정책 | 일관성 부족 |
| **운영 역량** | 전문 인력 부족 | 장애 대응 지연 |
| **확장성** | 수요 증가 시 개별 증설 | 비효율적 확장 |

### 2.3 문제점 및 개선 방향

#### 2.3.1 핵심 문제점

| 문제 영역 | 상세 내용 | 영향 |
|----------|----------|------|
| **예산 비효율** | 기관별 개별 인프라 구축으로 중복 투자 | 예산 낭비 |
| **자원 낭비** | 낮은 자원 활용률, 유휴 자원 과다 | 운영비 증가 |
| **보안 취약** | 일관된 보안 정책 부재, 데이터 혼입 우려 | 보안 사고 위험 |
| **역량 부족** | 인프라 구축 전문 인력 부족 | 품질 저하 |

## 3. 목표 업무 설계

### 3.1 개선 과제

| 개선 과제 | 세부 내용 |
|----------|----------|
| **멀티테넌시 아키텍처** | 테넌트 간 완전 격리 (네트워크/스토리지/컴퓨트/접근제어) |
| **고가용성 확보** | 장애 도메인 설계, 자동 복구, 이중화 |
| **망분리 체계** | 다계층 네트워크 분리, 마이크로세그멘테이션 |
| **스토리지 효율화** | 티어링, 백업, DR 설계 |
| **운영 자동화** | IaC, 중앙 집중 모니터링 |

### 3.2 목표 업무 설계

#### 3.2.1 컴퓨트

| 항목 | 설계 내용 |
|------|----------|
| **리소스 풀 구성** | KVM 하이퍼바이저 기반 VM 클러스터 |
| **장애 도메인 설계** | Host 단위 장애 격리, Anti-Affinity |
| **VM 배치 전략** | Affinity/Anti-Affinity 규칙 적용 |
| **리소스 쿼터** | 테넌트별 자원 제한 (vCPU, Memory) |
| **고가용성** | VM 자동 재시작, Live Migration |

#### 3.2.2 네트워크 및 보안

| 항목 | 설계 내용 |
|------|----------|
| **망분리 설계** | VLAN 기반 다계층 네트워크 분리 |
| **테넌트 간 격리** | 마이크로세그멘테이션으로 East-West 트래픽 제어 |
| **방화벽 정책** | 계층별 접근 제어 (DMZ → 업무망 → DB망) |
| **RBAC 설계** | 플랫폼관리자 / 테넌트관리자 / 사용자 역할 분리 |
| **암호화 체계** | 전송 구간(TLS), 저장 데이터(AES-256) |

#### 3.2.3 스토리지

| 항목 | 설계 내용 |
|------|----------|
| **분산 스토리지** | Ceph 기반 분산 저장, 데이터 복제 (replica 3) |
| **티어링 정책** | Hot(SSD) / Warm / Cold(HDD) 자동 이동 |
| **데이터 격리** | 테넌트별 스토리지 풀 분리, 쿼터 기반 용량 제한 |
| **백업 체계** | 정기 스냅샷, 증분 백업 |
| **DR 설계** | RPO/RTO 정의, 복제 전략 (설계 문서) |

### 4.3 설계 영향 요소 (Tier 1~4)

본 프로젝트는 단순 자원 생성이 아닌, **설계 근거가 있는 아키텍처**를 구현한다:

|------|------|----------|

| **Tier 2** | 데이터/처리 특성 | 트랜잭션, 실시간 처리, 대용량 데이터 |

| 암호화 | 전송/저장 데이터 암호화 | 데이터 유출 방지 |

### 7.4 포트폴리오 가치

| 관점 | 기대 효과 |
|------|----------|
| **설계 역량** | Tier 1~4 설계 영향 요소를 고려한 체계적 아키텍처 설계 경험 |
| **HCI 경험** | 에이블스택 기반 프라이빗 클라우드 구축 실무 경험 |
| **멀티테넌시** | 기관 간 격리 구조 설계 및 검증 |
| **IaC** | Terraform/Ansible 기반 인프라 자동화 |

## 8. 용어 정의

| 용어 | 정의 |
|------|------|
| **HCI** | Hyper-Converged Infrastructure, 컴퓨트/스토리지/네트워크를 단일 시스템으로 통합한 인프라 |
| **멀티테넌시** | 하나의 인프라에서 여러 조직(테넌트)을 격리하여 운영하는 구조 |
| **VLAN** | Virtual LAN, 물리적 네트워크를 논리적으로 분리하는 기술 |
| **마이크로세그멘테이션** | 워크로드 단위로 네트워크를 세분화하여 트래픽을 제어하는 보안 기법 |
| **Zero Trust** | "신뢰하지 않고 항상 검증"하는 보안 모델 |
| **RBAC** | Role-Based Access Control, 역할 기반 접근 제어 |
| **IaC** | Infrastructure as Code, 인프라를 코드로 정의하고 관리하는 방식 |
| **SDN** | Software Defined Networking, 소프트웨어로 네트워크를 제어하는 기술 |
| **SDS** | Software Defined Storage, 소프트웨어로 스토리지를 관리하는 기술 |
| **Ceph** | 오픈소스 분산 스토리지 시스템 |
| **DR** | Disaster Recovery, 재해 복구 |
| **RPO** | Recovery Point Objective, 데이터 손실 허용 시점 |
| **RTO** | Recovery Time Objective, 서비스 복구 목표 시간 |
| **HA** | High Availability, 고가용성 |
