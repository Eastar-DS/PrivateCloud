## 1. 사업 개요

### 1.1 추진 배경 및 필요성

#### 1.1.1 현행 문서 관리의 한계

A기관 내 각 부서는 업무 관련 문서를 개별적으로 관리하고 있으며, 부서 간 문서 공유 및 검색에 어려움을 겪고 있다.

| 문제 | 상세 내용 |
|------|----------|
| **체계적 분류 부재** | 과거 문서들이 체계적인 분류 없이 보관되어 있어 필요한 자료를 찾는 데 많은 시간 소요 |
| **접근 권한 설정 복잡** | 문서 공유 시 접근 권한 설정이 복잡하여 "전체 공개"로 설정하는 경우 빈번 |
| **보안 취약점** | 민감 문서가 불필요하게 노출될 위험 존재 |
| **검색 기능 미흡** | 파일명 검색만 가능, 내용 기반 검색 불가 |

### 1.2 서비스 목적 및 목표

#### 1.2.1 서비스 목적

> **에이블스택(ABLESTACK) HCI 기반 프라이빗 클라우드 위에 AI 문서 태깅 시스템을 구축하여, A기관 내 3개 부서가 문서를 안전하게 저장·공유·검색할 수 있는 환경을 제공한다.**

**핵심 목표:**

| 구분 | 목표 |
|------|------|
| **서비스** | AI 기반 문서 자동 태깅 및 검색 시스템 구축 |
| **인프라** | 부서 간 데이터 격리가 가능한 RBAC 기반 프라이빗 클라우드 구축 |
| **보안** | 문서별 접근 권한 관리 및 부서 간 데이터 유출 방지 |

#### 1.2.2 목표 서비스: AI 기반 문서 태깅

본 인프라가 지원하는 서비스로 **AI 기반 문서 자동 태깅**을 설정한다:

| 항목 | 내용 |
|------|------|
| **서비스 목적** | 업로드된 문서에 AI가 자동으로 주제/카테고리 태그 부여 |
| **기술 기반** | 텍스트 분류 모델 |
| **구현 수준** | 인프라 검증용 샘플 워크로드 (로컬 학습 모델 업로드) |
| **데이터 흐름** | 문서 업로드 → AI 태깅 (비동기) → 태그 기반 검색 |

#### 1.2.5 인프라 설계 목표

| 구분 | 목표 | 검증 방법 |
|------|------|----------|
| **부서 격리** | 3개 부서 데이터 완전 격리 (앱 레벨 RBAC) | 부서 간 데이터 접근 불가 테스트 |
| **문서 저장** | 부서별 스토리지 풀 분리, 쿼터 관리 | 스토리지 격리 검증 |
| **AI 서비스** | Spring AMQP Consumer 기반 비동기 처리, AI 서버 (Stateless) | 업로드 후 태깅 완료 확인 |
| **고가용성** | 핵심 서비스 장애 시 자동 복구 | Failover 시연 |
| **보안** | RBAC, 감사 로그 | 접근 제어 테스트 |

## 2. 현행 분석 및 문제점

### 2.1 현행 업무 현황

A기관은 현재 부서별로 문서를 개별 관리하고 있으며, 공유가 필요한 문서는 별도의 공용 시스템에 업로드하고 있다.

| 부서 | 현황 |
|------|------|
| **부서 A** | 자체 파일 서버에 문서 보관 |
| **부서 B** | 외부 클라우드 스토리지 사용 |
| **부서 C** | 공용 시스템 + 개별 PC 혼용 |

**공용 문서 시스템 현황:**
- 문서 업로드 시 수동으로 분류/태깅 필요
- 과거 문서는 연도별로만 정리되어 있음
- 검색 기능 미흡 (파일명 검색만 가능)

### 2.2 문제점 분석

#### 1) 문서 검색의 어려움
- 과거 문서들이 체계적으로 분류되어 있지 않음
- 필요한 자료를 찾기 위해 수기로 뒤져야 함
- 검색 시 파일명만 검색 가능, 내용 검색 불가

#### 2) 접근 권한 관리 미흡
- 문서 업로드 시 접근 권한 설정이 복잡함
- "어디까지 공개해야 할지 몰라서" 전체 공개로 설정하는 경우 빈번
- 민감 문서가 불필요하게 노출될 위험

#### 3) 부서 간 데이터 격리 부재
- 공용 시스템에서 부서 간 데이터가 혼재
- 타 부서 문서에 접근 가능한 경우 존재

#### 4) 분산된 인프라
- 부서별 개별 스토리지 운영으로 비용 낭비
- 통합 관리 불가

### 2.3 개선 방향

| 문제점 | 개선 방향 | 적용 기술 |
|--------|----------|----------|
| 문서 검색 어려움 | AI 기반 자동 태깅 | AI 태깅 모델, 비동기 처리 |
| 접근 권한 관리 미흡 | 문서별/부서별 권한 체계 | RBAC, DepartmentContext @Filter |
| 부서 간 격리 부재 | 앱 레벨 RBAC 기반 격리 | NFS 디렉토리 분리, Hibernate @Filter |
| 분산된 인프라 | 통합 프라이빗 클라우드 | HCI 기반 통합 인프라 |

| **ai-models** | 태깅 모델 저장 | Hot |

### 3.3 부서 격리 설계

#### 3.3.1 부서의 역할

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        부서 역할 정의                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   부서 = 문서 태깅 시스템을 이용하는 조직 단위 (앱 레벨 RBAC 격리)         │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  부서가 하는 일:                                                    │  │
│   │  • 문서를 NFS 디렉토리 기반 저장소에 업로드                          │  │
│   │  • AI 태깅 서비스를 통해 자동 태그 부여                             │  │
│   │  • 태그 기반 문서 검색 및 조회                                      │  │
│   │  • 문서별 접근 권한 설정                                            │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  부서가 갖는 것:                                                    │  │
│   │  • RBAC 기반 데이터 접근 권한 (DepartmentContext @Filter)           │  │
│   │  • NFS 디렉토리 기반 문서 저장 공간                                  │  │
│   │  • 공용 AI 서비스 접근 권한                                         │  │
│   │  • 검색 서비스 접근 권한                                            │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  부서 간 격리:                                                      │  │
│   │  ✗ 부서 A는 부서 B/C의 문서에 접근 불가 (RBAC)                      │  │
│   │  ✗ DB 조회 시 DepartmentContext @Filter 자동 적용                   │  │
│   │  ○ 공용 AI 서비스, 검색 서비스는 공유 (결과는 부서별 필터링)        │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3.3.2 부서 격리 원칙

| 격리 레이어 | 구현 방안 |
|------------|----------|
| **앱 레벨 RBAC** | Hibernate @Filter (DepartmentContext) 기반 데이터 격리 |
| **스토리지** | NFS 디렉토리 기반 부서별 문서 분리 |
| **접근 제어** | RBAC 4단계: platform-admin, dept-admin, dept-user, viewer |
| **데이터** | 메타데이터는 department_id로 필터링, 문서는 부서별 NFS 디렉토리에 저장 |
