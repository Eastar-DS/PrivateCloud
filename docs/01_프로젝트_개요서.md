## 1. 사업 개요

### 1.1 추진 배경 및 필요성

#### 1.1.1 정책 환경

2026년 재정경제부는 공공기관 인프라 활용 정책을 **개별 기관 중심에서 협업·확산 체계로 전환**한다고 발표했다.

| 정책 방향 | 세부 내용 |
|----------|----------|
| **공동 구축·활용** | 기관별 분산된 인프라를 통합하여 중복 투자 방지 |
| **데이터 주권 강화** | 민감 데이터의 외부 유출 방지를 위한 프라이빗 클라우드 인프라 필요 |
| **AI 공동 활용** | 고가의 GPU 자원을 다기관이 공유하여 비용 효율화 |
| **표준화 추진** | 공공기관에 적합한 표준 아키텍처 정립 필요 |

#### 1.1.2 기술적 필요성

기관별 개별 인프라 구축으로 인해 다음의 문제가 발생하고 있다:

| 과제 | 설명 |
|------|------|
| **멀티테넌시** | 여러 기관이 자원을 공유하면서도 완전한 격리 보장 필요 |
| **고가용성(HA)** | 단일 장애점(SPOF) 제거, 서비스 연속성 확보 |
| **망분리** | 공공기관 보안 요건에 따른 다계층 네트워크 분리 |
| **스토리지 효율화** | 대용량 데이터의 티어링, 백업, DR 설계 |
| **AI 자원 공유** | GPU 등 고가 자원의 효율적 공유 및 격리 |

#### 1.2.3 목표 상세

| 구분 | 목표 | 검증 방법 |
|------|------|----------|
| **멀티테넌시** | 3개 테넌트 완전 격리 | 테넌트 간 접근 불가 테스트 |
| **고가용성** | VM 장애 시 자동 복구 | Failover 시연 |
| **망분리** | 9개 이상 네트워크 존 분리 | 트래픽 격리 검증 |
| **스토리지** | Hot/Warm/Cold 티어링 | 데이터 이동 자동화 |
| **보안** | Zero Trust 아키텍처 | RBAC, 마이크로세그멘테이션 |
| **GPU** | 테넌트별 vGPU 할당, 공용 GPU 풀 | GPU 스케줄링 동작 |
| **운영** | IaC 기반 자동화 | Terraform/Ansible 배포 |

### 1.3 사업 범위

#### 1.3.1 In-Scope (수행 범위)

**인프라 설계 (핵심, 80%)**

| 영역 | 세부 항목 |
|------|----------|
| **컴퓨트** | 리소스 풀 설계, VM 배치 전략, 리소스 쿼터, GPU 가상화(vGPU), 고가용성(HA) |
| **네트워크** | 망분리 설계, VLAN 구성, 마이크로세그멘테이션, SDN 정책, 로드밸런서 |
| **스토리지** | 분산 스토리지 설계, 티어링 정책, 스냅샷/백업, DR 복제 |
| **보안** | Zero Trust 아키텍처, RBAC, 암호화 체계, 데이터 품질 게이트 |
| **운영** | IaC(Terraform/Ansible), 모니터링(Prometheus/Grafana), 로깅 |

**검증 워크로드 (부차적, 20%)**

| 항목 | 목적 |
|------|------|
| 샘플 AI 워크로드 | 로컬에서 학습한 모델 배포, 인프라 동작 검증 |
| 샘플 웹 애플리케이션 | 멀티테넌시 격리, 고가용성, 네트워크 분리 검증 |

#### 1.3.2 Out-Scope (제외 범위)

- 애플리케이션 개발 또는 고도화 (AI 모델 자체 개발 포함)
- 실제 대국민 서비스 제공
- GPU 하드웨어 실물 구매/설치 (설계만 수행)
- 퍼블릭 클라우드 또는 하이브리드 구성
- 상용 운영 환경 전환

### 1.4 기대 효과

#### 1.4.1 예산 절감 및 자원 효율화

| 항목 | 개별 구축 시 | 통합 구축 시 | 효과 |

- GPU, 고성능 스토리지 등 고가 자원의 공유 활용

#### 1.4.3 데이터 주권 확보

## 2. 현행 업무 분석 및 문제점

```
│   │ ─────── │        │ ─────── │        │ ─────── │        │ ─────── │    │

│   │ 스토리지 │        │ 스토리지 │        │ 스토리지 │        │ 스토리지 │    │
│   │ 네트워크 │        │ 네트워크 │        │ 네트워크 │        │ 네트워크 │    │

│   └─────────┘        └─────────┘        └─────────┘        └─────────┘    │
│        │                  │                  │                  │          │
```

| 구분 | 현황 |

```
│   │  • 장기 아카이브, 규정 준수 데이터                                  │  │
│   │  • Erasure Coding 적용 (저장 효율)                                  │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 클라우드 통합 관리 설계

#### 3.3.1 멀티테넌시 관리

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        멀티테넌시 격리 4계층                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  Layer 1: 네트워크 격리                                             │  │
│   │  • 테넌트별 전용 VLAN 할당                                          │  │
│   │  • 마이크로세그멘테이션으로 East-West 트래픽 차단                   │  │
│   │  • 공용 서비스 접근은 지정된 경로만 허용                            │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  Layer 2: 스토리지 격리                                             │  │
│   │  • 테넌트별 Ceph Pool 분리                                          │  │
│   │  • 쿼터 기반 용량 제한                                              │  │
│   │  • 테넌트별 암호화 키 분리                                          │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  Layer 3: 컴퓨트 격리                                               │  │
│   │  • 테넌트별 리소스 쿼터 (vCPU, Memory, VM 수)                       │  │
│   │  • GPU 사용 시 vGPU 쿼터 (메모리, 동시 작업 수)                     │  │
│   │  • Anti-Affinity로 VM 분산                                          │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  Layer 4: 접근제어 격리                                             │  │
│   │  • RBAC: 플랫폼관리자 / 테넌트관리자 / 사용자                       │  │
│   │  • 테넌트 A 관리자는 테넌트 B 리소스 접근 불가                      │  │
│   │  • 감사 로그 테넌트별 분리                                          │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3.3.2 통합 보안 체계

| 보안 영역 | 설계 내용 |
|----------|----------|
| **인증/인가** | SSO 연동, MFA (관리자 필수), RBAC |
| **Zero Trust** | 상시 검증, 최소 권한, 마이크로세그멘테이션 |
| **암호화** | 전송 (TLS 1.3), 저장 (AES-256), 키 관리 |
| **감사** | 모든 접근 로깅, 테넌트별 분리, 1년 보관 |
| **데이터 보호** | 품질 게이트, PII 익명화, 데이터 출처 추적 |

#### 3.3.3 자원 표준화 체계

| 표준화 항목 | 내용 |
|------------|------|
| **VM 템플릿** | 웹서버, 앱서버, DB, AI 학습, AI 추론 표준 이미지 |
| **GPU 프로파일** | vGPU 8GB/12GB/16GB 표준 프로파일 |
| **네트워크 정책** | 계층별 표준 방화벽 규칙 |
| **스토리지 클래스** | Hot/Warm/Cold 티어 정의 |
| **명명 규칙** | `{환경}-{테넌트}-{역할}-{번호}` |

#### 3.3.4 재해복구(DR) 체계

| DR 요소 | 설계 내용 |
|---------|----------|
| **RPO** | Tier 1 (핵심 DB): 15분 / Tier 2 (운영): 1시간 / Tier 3 (일반): 4시간 |
| **RTO** | Tier 1: 1시간 / Tier 2: 4시간 / Tier 3: 24시간 |
| **백업** | 일일 스냅샷 (7일 보관), 주간 전체 백업 (4주 보관) |
| **복제** | 핵심 데이터 동기 복제, 일반 데이터 비동기 복제 |
| **복구 절차** | Runbook 문서화, 분기별 복구 훈련 |

## 용어 정의

| 용어 | 정의 |
|------|------|
| **HCI** | Hyper-Converged Infrastructure, 컴퓨트/스토리지/네트워크를 단일 시스템으로 통합한 인프라 |
| **멀티테넌시** | 하나의 인프라에서 여러 조직(테넌트)을 격리하여 운영하는 구조 |
| **vGPU** | Virtual GPU, GPU를 여러 VM에 분할 할당하는 기술 |
| **MIG** | Multi-Instance GPU, NVIDIA A100/H100의 하드웨어 수준 GPU 분할 |
| **VLAN** | Virtual LAN, 물리적 네트워크를 논리적으로 분리하는 기술 |
| **마이크로세그멘테이션** | 워크로드 단위로 네트워크를 세분화하여 트래픽을 제어하는 보안 기법 |
| **Zero Trust** | "신뢰하지 않고 항상 검증"하는 보안 모델 |
| **RBAC** | Role-Based Access Control, 역할 기반 접근 제어 |
| **IaC** | Infrastructure as Code, 인프라를 코드로 정의하고 관리하는 방식 |
| **SDN** | Software Defined Networking, 소프트웨어로 네트워크를 제어하는 기술 |
| **SDS** | Software Defined Storage, 소프트웨어로 스토리지를 관리하는 기술 |
| **Ceph** | 오픈소스 분산 스토리지 시스템 |
| **DR** | Disaster Recovery, 재해 복구 |
| **RPO** | Recovery Point Objective, 데이터 손실 허용 시점 |
| **RTO** | Recovery Time Objective, 서비스 복구 목표 시간 |
| **HA** | High Availability, 고가용성 |
| **데이터 품질 게이트** | 공용 학습에 사용될 데이터의 품질 검증 및 익명화 처리 계층 |
| **모델 레지스트리** | 학습된 AI 모델의 버전 관리 및 배포 저장소 |
