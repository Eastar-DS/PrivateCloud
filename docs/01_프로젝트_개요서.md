## 1. 사업 개요

### 1.1 추진 배경 및 필요성

#### 1.1.1 지자체 AI 서비스 도입의 한계

지방자치단체에서 AI 서비스(예: 민원 자동 분류, 문서 요약 등)를 도입하려면 다음과 같은 어려움에 직면한다:

| 문제 | 상세 내용 |
|------|----------|
| **GPU 인프라 부재** | AI 학습/추론에 필요한 GPU 서버를 개별 구매하기 어려움 |
| **전문 인력 부족** | AI 인프라 구축 및 운영 역량 부족 |
| **데이터 주권 우려** | 민원 데이터를 외부 클라우드에 올리기 어려움 (개인정보 포함) |
| **중복 투자** | 기관마다 유사한 AI 서비스를 위해 개별 인프라 구축 |

### 1.2 서비스 목적 및 목표

#### 1.2.1 서비스 목적

> **다수의 지자체가 공용 AI 서비스(민원 자동 분류)를 안전하게 이용할 수 있는 멀티테넌트 AI 플랫폼 인프라를 설계한다.**

**핵심 개념:**
- **테넌트** = AI 서비스를 이용하는 지자체 (수원시, 성남시, 용인시 민원담당)
- **테넌트의 역할** = 민원 데이터 제공, AI 분류 결과 활용
- **플랫폼의 역할** = 데이터 수집 → 학습 → 추론 API 제공

#### 1.2.2 목표 서비스: 민원 자동 분류

본 인프라가 지원하는 대표 서비스로 **민원 자동 분류**를 설정한다:

| 항목 | 내용 |
|------|------|
| **서비스 목적** | 민원 접수 시 담당 부서 자동 분류 |
| **기술 기반** | 텍스트 분류 모델 (BERT 기반) |
| **구현 수준** | 인프라 검증용 샘플 워크로드 |
| **데이터 흐름** | 3개 지자체 민원 데이터 → 품질 게이트 → 공용 모델 학습 → 추론 API |

#### 1.2.5 인프라 설계 목표

| 구분 | 목표 | 검증 방법 |
|------|------|----------|
| **테넌트 격리** | 3개 테넌트 데이터 완전 격리 | 테넌트 간 데이터 접근 불가 테스트 |
| **GPU 공유** | 공용 GPU 풀 + 테넌트별 vGPU 쿼터 | GPU 스케줄링 동작 검증 |
| **데이터 품질** | 품질 게이트 통과 데이터만 학습 반영 | 익명화, 검증 프로세스 동작 |
| **고가용성** | 핵심 서비스 장애 시 자동 복구 | Failover 시연 |
| **보안** | Zero Trust, RBAC | 접근 제어 테스트 |

### 1.3 사업 범위

#### 1.3.1 In-Scope (수행 범위)

**핵심: AI 플랫폼 인프라 설계 (100%)**

| 계층 | 설계 항목 |
|------|----------|
| **계층 1: 테넌트 데이터** | 테넌트별 VLAN 격리, 전용 스토리지 Pool, 데이터 접근 제어 |
| **계층 2: 품질 게이트** | 데이터 검증 환경, 익명화 처리 설계, 승인 프로세스 |
| **계층 3: 공용 AI** | GPU 풀 설계, 학습 스케줄링, 모델 레지스트리, 고속 스토리지 |
| **계층 4: 서비스** | API Gateway, 추론 서버, 로드밸런서, DMZ 네트워크 |
| **공통** | 모니터링, 로깅, IaC, HA 설계 |

**검증 워크로드 (인프라 동작 확인용)**

| 항목 | 목적 |
|------|------|
| 샘플 민원 분류 모델 | 4계층 데이터 흐름 검증 |
| 테넌트 격리 테스트 | 네트워크/스토리지 격리 검증 |

#### 1.3.2 Out-Scope (제외 범위)

| 제외 항목 | 사유 |
|----------|------|
| AI 모델 개발 | 인프라 프로젝트, 서비스 개발 아님 |
| 실제 민원 시스템 연동 | 검증 환경, 운영 환경 아님 |
| 테넌트별 업무 시스템 (웹/앱/DB) | AI 플랫폼 인프라에 집중 |
| 퍼블릭 클라우드 연동 | 프라이빗 클라우드 단독 구성 |

### 1.4 기대 효과

#### 1.4.1 비용 효율화

| 항목 | 개별 구축 | 공용 플랫폼 | 효과 |
|------|----------|-----------|------|
| GPU 서버 | 기관당 1대 이상 | 공용 풀 공유 | 70% 이상 절감 |
| 운영 인력 | 기관별 필요 | 통합 운영 | 인력 효율화 |
| 활용률 | 20~30% | 70% 이상 | 3배 향상 |

#### 1.4.2 데이터 주권 확보

| 보안 요소 | 구현 방안 |
|----------|----------|
| **데이터 격리** | 테넌트별 네트워크/스토리지 분리 |
| **익명화** | 품질 게이트에서 PII 마스킹 |
| **접근 통제** | RBAC, 테넌트별 권한 분리 |
| **프라이빗** | 외부 클라우드 없이 자체 인프라 |

#### 1.4.3 AI 서비스 접근성 향상

- GPU 인프라가 없는 중소 지자체도 AI 서비스 이용 가능
- 표준화된 API로 일관된 서비스 품질

## 2. 현행 분석 및 문제점

### 2.1 현행 AI 서비스 환경

지자체에서 AI 서비스를 도입하려면 다음 중 하나를 선택해야 한다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        현행 AI 서비스 도입 방식                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   [방식 1] 자체 구축                    [방식 2] 외부 클라우드 활용          │
│                                                                             │
│   ┌─────────────────────┐              ┌─────────────────────┐             │
│   │  지자체 자체 DC      │              │  퍼블릭 클라우드     │             │
│   │  ─────────────────  │              │  (AWS, Azure 등)    │             │
│   │  • GPU 서버 구매     │              │  ─────────────────  │             │
│   │  • 인프라 구축       │              │  • GPU 인스턴스      │             │
│   │  • 전문 인력 채용    │              │  • AI 서비스 API     │             │
│   └─────────────────────┘              └─────────────────────┘             │
│                                                                             │
│   문제점:                               문제점:                              │
│   • 높은 초기 비용                      • 데이터 주권 우려                   │
│   • 낮은 활용률                         • 민감 정보 외부 전송                │
│   • 전문 인력 부족                      • 지속적인 비용 발생                 │
│   • 유지보수 부담                       • 종속성 (Lock-in)                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 문제점 분석

| 문제 | 상세 내용 | 영향 |
|------|----------|------|
| **높은 진입 장벽** | GPU 인프라 구축에 수억 원 소요 | 중소 지자체 AI 도입 불가 |
| **데이터 주권** | 민원 데이터에 개인정보 포함, 외부 전송 불가 | 퍼블릭 클라우드 활용 제한 |
| **중복 투자** | 유사 서비스를 위해 기관별 개별 구축 | 예산 낭비 |
| **낮은 활용률** | 자체 구축 시 GPU 활용률 20~30% | 자원 낭비 |

## 3. 목표 시스템 설계

### 3.1 AI 플랫폼 4계층 아키텍처

#### 3.1.1 계층별 설계 개요

| 계층 | 역할 | 주요 인프라 |
|------|------|-----------|
| **계층 4: 서비스** | 추론 API 제공 | API Gateway, 추론 서버 (vGPU), LB |
| **계층 3: 공용 AI** | 모델 학습/관리 | GPU 풀 (Passthrough), 모델 레지스트리, 고속 스토리지 |
| **계층 2: 품질 게이트** | 데이터 검증/익명화 | 처리 VM, 스테이징 스토리지 |
| **계층 1: 테넌트 데이터** | 원본 데이터 격리 저장 | 테넌트별 VLAN, 전용 Pool |

#### 3.1.2 계층별 인프라 매핑

| 계층 | 컴퓨트 | 네트워크 | 스토리지 | 보안 |
|------|--------|---------|---------|------|
| **계층 4** | 추론 서버 (vGPU) | DMZ (VLAN 10) | 모델 캐시 | API 인증, Rate Limit |
| **계층 3** | 학습 서버 (GPU Passthrough) | AI 내부망 (VLAN 50) | 학습 데이터 (Hot Tier) | 승인된 데이터만 접근 |
| **계층 2** | 품질 게이트 VM | AI 내부망 (VLAN 50) | 스테이징 영역 | 익명화, 검증 |
| **계층 1** | 데이터 업로드 VM (최소) | 테넌트 VLAN (20,21,22) | 테넌트 전용 Pool | 테넌트 간 완전 격리 |

### 3.2 인프라 설계 개요

> **상세 내용은 별도 설계서 참조**

#### 3.2.1 컴퓨트 설계 ([04_컴퓨트_설계서.md](04_컴퓨트_설계서.md))

| 설계 항목 | 내용 |
|----------|------|
| **GPU 가상화** | 학습: Passthrough (전용), 추론: vGPU (공유) |
| **GPU 풀** | 공용 GPU 클러스터, Fair-Share 스케줄링 |
| **리소스 쿼터** | 테넌트별 vGPU 할당량 제한 |
| **고가용성** | 추론 서버 HA, 학습 서버 체크포인트 |

#### 3.2.2 네트워크/보안 설계 ([05_네트워크_보안_설계서.md](05_네트워크_보안_설계서.md))

| VLAN ID | 네트워크 존 | 용도 |
|---------|-----------|------|
| **10** | DMZ | API Gateway, 추론 서비스 |
| **20** | 테넌트 A | 수원시 데이터 영역 |
| **21** | 테넌트 B | 성남시 데이터 영역 |
| **22** | 테넌트 C | 용인시 데이터 영역 |
| **50** | AI 내부망 | 학습, 품질 게이트 |
| **55** | AI 서비스망 | 추론 서비스 |
| **60** | AI 스토리지망 | 학습 데이터 I/O |
| **100** | 관리망 | 모니터링, SSH |

#### 3.2.3 스토리지 설계 ([06_스토리지_설계서.md](06_스토리지_설계서.md))

| 스토리지 Pool | 용도 | 티어 |
|--------------|------|------|
| **tenant-a-pool** | 수원시 원본 데이터 | Warm |
| **tenant-b-pool** | 성남시 원본 데이터 | Warm |
| **tenant-c-pool** | 용인시 원본 데이터 | Warm |
| **ai-training-pool** | 학습 데이터 (승인 후) | Hot |
| **model-registry-pool** | 학습된 모델 저장 | Hot |
| **staging-pool** | 품질 게이트 처리 영역 | Warm |

### 3.3 테넌트 격리 설계

#### 3.3.1 테넌트의 역할 (AI 플랫폼 중심)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        테넌트 역할 정의                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   테넌트 = AI 서비스 이용자 (데이터 제공자)                                 │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  테넌트가 하는 일:                                                  │  │
│   │  • 민원 데이터를 전용 스토리지에 업로드                             │  │
│   │  • 데이터가 품질 게이트를 통과하면 공용 학습에 기여                 │  │
│   │  • 추론 API를 호출하여 민원 분류 결과 활용                          │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  테넌트가 갖는 것:                                                  │  │
│   │  • 전용 네트워크 (VLAN)                                             │  │
│   │  • 전용 스토리지 (Ceph Pool)                                        │  │
│   │  • 데이터 업로드/관리용 최소 VM (선택)                              │  │
│   │  • API 접근 토큰                                                    │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  테넌트가 갖지 않는 것 (범용 클라우드와 차이):                      │  │
│   │  ✗ 웹서버, 앱서버 (업무 시스템 마이그레이션 아님)                   │  │
│   │  ✗ 자체 DB 서버 (공용 메타데이터 DB 사용)                           │  │
│   │  ✗ 자체 GPU (공용 GPU 풀 사용)                                      │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3.3.2 테넌트 격리 원칙

| 격리 레이어 | 구현 방안 |
|------------|----------|
| **네트워크** | 테넌트별 전용 VLAN, 마이크로세그멘테이션 |
| **스토리지** | 테넌트별 Ceph Pool 분리, 쿼터 |
| **접근 제어** | RBAC, 테넌트 간 리소스 접근 차단 |
| **데이터** | 원본 데이터는 테넌트 영역에만 존재, 품질 게이트 통과 후 익명화 |
