## 2. 팀별 역할 정의

### 2.1 컴퓨트/개발 팀 (2명)

#### 2.1.1 역할 개요

AI 문서 태깅 플랫폼의 컴퓨트 계층을 담당하며, KoBERT 기반 태깅 서비스 배포를 통해 인프라 동작을 검증한다.
테넌트(부서)는 문서 업로드 및 검색 기능을 사용하며, AI 자동 태깅 서비스는 공용 플랫폼에서 제공한다.

> **참고**: AI 추론은 CPU 기반 KoBERT 모델을 사용하며, 로컬에서 파인튜닝된 모델을 업로드하여 추론 서비스를 검증한다.

#### 2.1.2 담당 영역

| 영역 | 세부 내용 |
|------|----------|
| **리소스 풀 설계** | 클러스터 풀 구성, 오버커밋 정책 (CPU 2:1, Memory 1:1) |
| **VM 배치 전략** | Affinity/Anti-Affinity 규칙, 장애 도메인 분리 |
| **리소스 쿼터** | 테넌트별 API 사용량 제한, 리소스 쿼터 |
| **고가용성(HA)** | VM 자동 재시작, Live Migration 설정 |
| **VM 템플릿** | 웹 서버 VM, App 서버 VM, AI Worker VM, Model Train VM, DB Primary/Replica VM |
| **검증 워크로드** | 문서 자동 태깅 AI 모델 배포 (KoBERT CPU 추론), 서비스 흐름 검증 |

#### 2.1.3 AI 모델 서빙 담당 상세

> **범위**: KoBERT 기반 텍스트 분류 모델을 CPU에서 추론하여 문서 자동 태깅 서비스를 검증한다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    컴퓨트/개발 팀 AI 모델 서빙 범위                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. 모델 준비                                                              │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • KoBERT 기반 텍스트 분류 모델 (.pt)                              │  │
│   │  • 로컬에서 파인튜닝 후 플랫폼에 업로드                             │  │
│   │  • CPU 기반 추론 (PyTorch CPU 버전)                                │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   2. 추론 서비스 구성                                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 모델 서빙 VM: KoBERT 모델 로드, 추론 API 제공                   │  │
│   │  • REST API: POST /api/v1/predict                                  │  │
│   │  • 지연 시간 목표: <500ms (CPU 추론)                               │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   3. VM 템플릿 (8 VM 구성)                                                 │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 웹 서버 VM x2: nginx (Reverse Proxy), HA 구성                  │  │
│   │  • App 서버 VM x2: Spring Boot (REST API), HA 구성                │  │
│   │  • AI Worker VM x1: KoBERT 추론 + OCR 처리 (Docker Python)        │  │
│   │  • Model Train VM x1: 주기적 모델 파인튜닝 (Docker Python)        │  │
│   │  • DB Primary VM x1: MariaDB (Read/Write)                         │  │
│   │  • DB Replica VM x1: MariaDB (Read Only), 복제 및 읽기 분산       │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 2.1.4 필요 역량

- KVM/QEMU 하이퍼바이저 이해
- 가상화 기술 (VM, 컨테이너) 기초
- Linux 시스템 관리
- **AI/ML 프레임워크 기초 (PyTorch)** - 모델 배포 수준
- **KoBERT 모델 활용** - 텍스트 분류
- 모델 서빙 및 API 개발 능력

#### 2.1.5 학습 키워드

```
- KVM 하이퍼바이저
- 리소스 풀 (Resource Pool)
- Affinity / Anti-Affinity
- 장애 도메인 (Fault Domain)
- Live Migration
- VM 템플릿
- Cloud-init
- KoBERT, PyTorch (CPU 추론)
- 모델 서빙 (FastAPI)
- 비동기 처리 (Celery, RabbitMQ)
```

### 2.2 네트워크/보안 팀 (2명)

#### 2.2.1 역할 개요

네트워크 계층의 설계 및 보안 아키텍처를 담당하며, 멀티테넌시 환경에서의 격리, 접근제어, 감사 로깅을 구현한다.

> **참고**: 사용자 인증(로그인)은 구현하지 않으며, 기존 내부 인증 체계를 전제로 사용자 정보를 받아 저장하는 방식으로 구현한다.

#### 2.2.2 담당 영역

| 영역 | 세부 내용 |
|------|----------|
| **망분리 설계** | 네트워크 존 정의, VLAN 할당 계획, IP 대역 설계 |
| **마이크로세그멘테이션** | 테넌트 간 격리, 계층 간 통신 정책 |
| **방화벽 정책** | 계층별 접근 제어 규칙 정의 |
| **로드밸런서** | L4/L7 로드밸런싱, URL 기반 라우팅 |
| **RBAC 설계** | 역할 정의 (플랫폼관리자/테넌트관리자/사용자), 권한 매핑 |
| **사용자 정보 관리** | 기존 인증 체계 연동, 사용자 정보 저장 및 관리 |
| **암호화 체계** | TLS 1.3 설정, 저장 데이터 AES-256 암호화 정책 |
| **감사 로깅** | 접근 로그, 감사 추적, 테넌트별 로그 분리 |

#### 2.2.3 접근 제어 및 사용자 관리 상세

```
┌─────────────────────────────────────────────────────────────────────────────┐
│               네트워크/보안 팀 접근 제어 설계 범위                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   사용자 인증은 기존 내부 인증 체계를 전제로 하며,                           │
│   본 시스템은 사용자 정보를 받아 권한 관리 및 감사에 활용한다.               │
│                                                                             │
│   1. 인증 전제 조건                                                          │
│   ┌─────────────────────────────────────────────────────────────────────┐  │

│   │  • 로그인 기능 미구현 (인증은 기존 시스템에서 처리)                  │  │

│   2. 사용자 정보 관리                                                        │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 사용자 ID, 소속 부서, 역할 정보 저장                             │  │
│   │  • 테넌트(부서)별 사용자 매핑                                       │  │
│   │  • 권한 정보 관리                                                   │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   3. RBAC (역할 기반 접근 제어)                                              │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 플랫폼 관리자: 전체 시스템 관리, 테넌트 생성/삭제               │  │
│   │  • 테넌트 관리자: 부서 내 사용자 관리, 문서 관리                    │  │
│   │  • 일반 사용자: 문서 업로드, 검색, 조회                             │  │

│                                                                             │
│   4. 감사 로깅                                                               │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 모든 API 호출 기록                                               │  │
│   │  • 문서 접근 이력 추적                                              │  │
│   │  • 테넌트별 로그 분리 저장                                          │  │
│   │  • 감사 로그 보존 정책                                              │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 스토리지 팀 (2명)

#### 2.3.1 역할 개요

AI 문서 태깅 플랫폼의 스토리지 계층을 담당하며, 테넌트별 문서 저장과 모델 저장소를 통합 설계한다.
테넌트는 문서를 업로드하며, 문서 원본 및 메타데이터가 분리 저장된다.

#### 2.3.2 담당 영역

| 영역 | 세부 내용 |
|------|----------|
| **분산 스토리지 설계** | Ceph 클러스터 구성, 복제 정책 (replica 3) |
| **티어링 정책** | Hot (NVMe) / Warm (SATA SSD) / Cold (HDD) 정의, 자동 이동 규칙 |
| **테넌트 격리** | 테넌트별 Ceph Pool 분리, 쿼터 설정 |
| **문서 스토리지** | 문서 원본 저장, 메타데이터/태그 분리 저장 |
| **모델 저장소** | 모델 레지스트리 전용 Pool, 버전 관리 |
| **스냅샷/백업** | 일일/주간 스냅샷 정책, 백업 자동화 |
| **DR 설계** | RPO/RTO 정의, 복제 전략 (설계 문서) |
| **성능 최적화** | 캐시 설정, I/O 튜닝 |

#### 2.3.3 문서 스토리지 설계 상세

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    스토리지 팀 문서 스토리지 설계 범위                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   문서 태깅 플랫폼은 문서 원본과 메타데이터를 분리 저장한다.                 │
│   스토리지 팀은 다음 영역을 설계한다:                                       │
│                                                                             │
│   1. 문서 원본 스토리지                                                     │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 테넌트별 Pool 분리: tenant-a-docs, tenant-b-docs, tenant-c-docs  │  │
│   │  • 문서 형식: PDF, DOCX, TXT, HWP 등                               │  │
│   │  • 쿼터 설정: 테넌트별 저장 용량 제한                               │  │
│   │  • 접근 제어: 테넌트 간 문서 접근 차단                              │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   2. 메타데이터 스토리지                                                    │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 메타데이터 DB: 문서 ID, 제목, 업로드 일시, 소유자                │  │
│   │  • 태그 정보: AI 자동 태깅 결과, 수동 태그                          │  │
│   │  • 검색 인덱스: 태그 기반 검색을 위한 인덱싱                         │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   3. 모델 저장소 (Model Registry)                                          │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 전용 Ceph Pool: model-registry-pool                             │  │
│   │  • 버전 관리: 모델 버전별 스냅샷                                   │  │
│   │  • 빠른 배포: 모델 서빙 서버에서 모델 로드 최적화                   │  │
│   │  • 사전 학습 모델 저장: 로컬에서 학습 후 업로드된 모델              │  │
```


### 4.2 협업 상세

| 협업 영역 | 관련 팀 | 협업 내용 |
|----------|--------|----------|
| **VM-네트워크 연결** | 컴퓨트 + 네트워크 | VM의 네트워크 인터페이스 설정, VLAN 할당 |
| **VM-스토리지 연결** | 컴퓨트 + 스토리지 | VM 디스크 볼륨 연결, 스토리지 풀 매핑 |
| **스토리지 네트워크** | 네트워크 + 스토리지 | 스토리지 전용 VLAN, 고속 네트워크 설정 |
| **보안 정책 적용** | 네트워크 + 컴퓨트 | 테넌트별 보안 그룹, VM 방화벽 규칙 |
| **태깅 워커 스토리지** | 컴퓨트 + 스토리지 | 태깅 워커 VM의 문서 스토리지 연결 |
| **API 인증 연동** | 컴퓨트 + 네트워크 | API Gateway의 사용자 정보 처리 |
| **모니터링 통합** | 전체 | ABLESTACK Mold 기본 모니터링 활용 |
| **IaC 통합** | 전체 | Terraform 모듈 통합, Ansible 플레이북 연계 |

### 4.3 서비스 계층별 담당

| 계층 | 담당 팀 | 역할 |
|------|--------|------|
| **웹/API 계층** | 컴퓨트 (웹 서버, API Gateway), 네트워크 (LB) | 사용자 인터페이스, API 라우팅 |
| **비즈니스 계층** | 컴퓨트 (태깅 워커), 네트워크 (접근 제어) | 문서 처리, 태깅 로직 |
| **AI 서비스 계층** | 컴퓨트 (모델 서빙), 스토리지 (모델 저장소) | AI 추론, 모델 관리 |
| **데이터 계층** | 스토리지 (문서/메타데이터), 네트워크 (격리) | 데이터 저장 및 검색 |
